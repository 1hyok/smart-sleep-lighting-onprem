## 1. 최종 시스템 설계 (수집 경로)

프로젝트 3의 엣지 노드는 조도 센서값을 **로컬 Mosquitto 브로커**(`mqtt://localhost:1883`,
user/pass)로 발행했다. 프로젝트 6에서는 발행 대상을 **AWS IoT Core**(`mqtts://...:8883`,
X.509 mTLS)로 전환하고, **토픽·페이로드 계약은 온프레미스 그대로 유지**하여 IoT Rule SQL 과
백엔드(임형택) 구독이 변경 없이 붙도록 했다. 데이터 흐름은 다음과 같다.

```
[YL-40 조도센서] --I2C--> [RPi: sensor.js 노이즈필터(메디안+이동평균)]
        --> [iotClient.js : mqtts 8883, X.509 상호 TLS, QoS 1]
        --(publish)--> AWS IoT Core 메시지 브로커
              · home/sensor/illuminance   (조도 텔레메트리)
              · home/edge/status          (LWT online/offline, retain)
        --> [IoT Rules Engine — layers/ingestion.yaml]
              ├ Rule A: SELECT ... FROM 'home/sensor/illuminance' WHERE source='sensor'
              │     --> DynamoDB illuminance-readings(이력) + illuminance-latest(최신)  ※이준혁 테이블
              └ Rule B: SELECT * FROM 'home/edge/status'
                    --> DeviceStatus Lambda(임형택, 선택) + CloudWatch Logs(가용성 모니터링)

[AWS IoT Core] <--Device Shadow desired/reported--> [RPi: shadow.js → light.js GPIO 실행]
                       ▲ desired 갱신(조명 명령) = 백엔드 Lambda  ※임형택 경계
```

설계의 핵심 결정(프로젝트 5 §2.1 확정 → 본 프로젝트 구현)은 다음 다섯 가지다.

1. **인증 — X.509 상호 TLS(mTLS):** 디바이스별 클라이언트 인증서/개인키로 IoT Core(8883)에
   접속하고 Amazon Root CA 로 서버 인증서를 검증(`rejectUnauthorized:true`). 평문 MQTT(1883)·
   익명 접속을 허용하지 않아 위장 디바이스의 메시지 주입을 차단한다.
2. **토픽 계약 유지:** `home/sensor/illuminance`·`home/edge/status` 를 그대로 사용해 IoT Rule
   SQL·DeviceStatus Lambda·DynamoDB 매핑이 추가 변환 없이 일치한다(다중 디바이스 격리를 위한
   `iot/<thing>/...` 네임스페이스 전환은 후속 과제).
3. **전송 신뢰성 — QoS 1:** 손실 방지(QoS 0 대비)·오버헤드 절감(QoS 2 대비). 재전송 중복은
   수신 측이 `(device_id, recorded_at)` 키로 idempotent 처리(DynamoDB PK/SK).
4. **가용성 — LWT:** 비정상 단절 시 IoT Core 가 `home/edge/status` 에 `offline` 을 자동 retain
   발행. 정상 접속/종료 시 `online`/`offline` 을 명시적으로 retain 갱신한다.
5. **명령 채널 — Device Shadow:** 고빈도 조도는 일반 토픽, 조명 명령은 Shadow(desired→reported)로
   분리. 엣지가 desired 를 구독해 **로컬 GPIO 로 직접 램프를 제어**하므로 연결이 끊겨도 재접속 시
   마지막 명령을 복구·동기화한다.

## 2. 구현 결과 (소스 코드 설명)

```
aws/
├── layers/ingestion.yaml         ★ IoT Rule 2개 + 최소권한 IAM 2종 + CloudWatch 로그 그룹
├── samconfig-ingestion.toml        sam deploy --config-file samconfig-ingestion.toml --config-env ingestion
└── src/ingestion/
    ├── edge/                      라즈베리파이 엣지 노드 (Node.js)
    │   ├── index.js              진입점: 센서→발행 + Shadow 연결 + graceful shutdown
    │   ├── iotClient.js          ★ mqtts mTLS, QoS1, LWT online/offline, 지수 백오프
    │   ├── shadow.js             ★ Device Shadow delta 구독 → 조명 명령 수신 → reported
    │   ├── light.js              ★ GPIO PWM actuation (SLEEP/WAKE 단계 램프)
    │   ├── sensor.js              YL-40(PCF8591) I2C 조도 + 노이즈필터 (프로젝트 3 재사용)
    │   ├── config.js · logger.js · .env.example · package.json
    └── provisioning/
        ├── iot-policy.json        ★ 최소권한 IoT Policy (정책 변수)
        ├── provision.sh           ★ Thing/Policy/인증서 발급·attach·Root CA (멱등)
        └── teardown.sh            리소스 정리
```

### 2.1 `edge/iotClient.js` — IoT Core 발행 클라이언트
온프렘 `mqttClient.js`(로컬 Mosquitto)를 IoT Core 용으로 진화시킨 모듈. `mqtt.js` 의
`protocol:'mqtts'` + `{key,cert,ca}` 로 X.509 상호 TLS 를 수립하고 `will` 로 LWT 를 등록한다.
정상 접속 시 `home/edge/status` 에 `online` 을 retain 발행하여 구독자가 어느 시점에 붙어도 1건
read 로 가용 여부를 안다. 영구 단절 시 지수 백오프(2초→최대 60초)로 재연결한다.

### 2.2 `edge/shadow.js` + `edge/light.js` — 조명 명령 수신·실행
Classic Shadow 토픽(`update/delta`, `get/accepted`)을 구독해 백엔드가 내린 `desired`(조명 명령)를
수신하면 **엣지 로컬에서 `light.js` 가 GPIO PWM 으로 램프를 실행**한다. SLEEP(80→60→40→20→0,
~30분)·WAKE(20→50→80→100, ~15분) 단계 테이블은 온프렘 `lightingExecutor.js` 를 엣지로 이전한
것이다. 실행 후 `reported` 에 `running`/`completed`/`cancelled` 상태를 반영한다. 메시지 핸들러는
1회만 바인딩하고 **매 (재)연결마다 재구독 + `shadow/get`** 으로 오프라인 동안 누락된 명령을 복구한다.

### 2.3 `edge/index.js` — 발행 페이로드 contract
주기마다 센서값을 읽어 아래 스키마로 발행한다. 이 스키마가 수신 측(이준혁/임형택)과의 인계 contract 다.
```json
// home/sensor/illuminance  (QoS 1, retain false)
{ "deviceId": "rpi-edge-bedroom-01", "value": 712.3, "raw": 198,
  "source": "sensor", "unit": "lux_estimate", "timestamp": "ISO8601" }
// home/edge/status  (QoS 1, retain TRUE, LWT)
{ "deviceId": "rpi-edge-bedroom-01", "status": "online|offline",
  "timestamp": "ISO8601", "reason": "graceful_shutdown|unexpected_disconnect" }
```

### 2.4 `provisioning/iot-policy.json` — 최소권한 (§6 보안)
`${iot:Connection.Thing.ThingName}` 정책 변수로 Connect/Shadow 를 자기 Thing 으로 한정한다. 한
디바이스 인증서가 유출되어도 다른 디바이스의 Shadow 에 접근할 수 없다.

### 2.5 `provisioning/provision.sh` — 디바이스 신원 프로비저닝
AWS CLI 로 ① Thing 생성 ② 정책 생성(region/account 치환) ③ X.509 인증서·키 발급 ④ 인증서↔정책↔
Thing attach ⑤ Amazon Root CA 다운로드 ⑥ 데이터 엔드포인트 안내. **멱등** — 기존 활성 인증서가
있으면 재사용해 재실행 시 고아 인증서가 쌓이지 않는다(`FORCE_NEW_CERT=true` 로 로테이션).

### 2.6 `layers/ingestion.yaml` — 라우팅·모니터링 (Storage Export Import)
- **Rule A** `SELECT deviceId AS device_id, timestamp AS recorded_at, value, source, raw,
  (floor(timestamp()/1000)+2592000) AS ttl FROM 'home/sensor/illuminance' WHERE source='sensor'`
  → DynamoDB `illuminance-readings`(이력) + `illuminance-latest`(최신값) **이중 PutItem**.
  Lambda 를 우회해 비용을 낮춘다(보고서 §4.2). `WHERE source='sensor'` 로 mock 표본은 제외.
- **Rule B** `SELECT * FROM 'home/edge/status'` → DeviceStatus Lambda(임형택, 선택) + CloudWatch
  Logs(가용성 모니터링). 두 Rule 모두 실패 시 ErrorAction 으로 오류 로그 그룹에 적재.
- IAM 은 IoT→DynamoDB(`PutItem`, 두 테이블만)·IoT→CloudWatch(지정 로그 그룹만) **최소권한** 2종.

## 4. 시연·캡쳐 가이드 (보고서 화면 캡쳐 / YouTube 시연용)

1. **로컬 dry-run 실행 로그** — `cd aws/src/ingestion/edge && MOCK_IOT=true MOCK_SENSOR=true npm start`
   → 부팅·online 발행·조도 주기 발행(QoS 1)·SIGTERM 시 offline 발행 로그.
2. **프로비저닝** — `provisioning/provision.sh` 실행 출력(인증서 발급·attach·엔드포인트).
3. **IoT 콘솔 MQTT 테스트** — `home/sensor/illuminance` 구독 화면에 실시간 조도 메시지 도착.
4. **IoT 콘솔 Security** — 발급된 인증서와 최소권한 정책.
5. **DynamoDB** — `illuminance-readings`/`illuminance-latest` 테이블에 적재된 항목.
6. **CloudWatch Logs** — `/smartsleep/dev/iot/device-status` 에 online/offline 라우팅.
7. **Device Shadow** — 콘솔에서 desired(`{"routine":"sleep"}`) 갱신 → 엣지 로그에 조명 루틴 실행 +
   reported 갱신.

## 5. 결론 — 정일혁 수행 소감

- **느낀 점:** 프로젝트 3의 로컬 MQTT 발행 코드에서 연결 계층(`iotClient.js`)만 mqtts + X.509 로
  교체하고 토픽/페이로드 계약을 유지하니, 하류 팀원(이준혁 DynamoDB·임형택 Lambda) 작업과 독립적으로
  내 파트를 완성·검증할 수 있었다. "역할 경계를 토픽과 Rule 액션이라는 명시적 contract 로 끊어 두면"
  통합이 한 줄(액션 추가)로 끝난다는 점을 체감했다.
- **배운 점:** X.509 mTLS 의 키/인증서/CA 3요소가 각각 무엇을 검증하는지, IoT Policy 변수로
  디바이스 단위 최소권한을 격리하는 법, IoT Rule 이 Lambda 없이 DynamoDB 에 직접 쓰는 SQL 매핑,
  Device Shadow 로 오프라인 복구가 되는 명령 채널을 코드 수준에서 이해했다.
- **아쉬운 점:** 시간/비용 제약으로 실제 AWS 배포·부하 시연은 시연 단계로 남겼고, 다중 디바이스
  토픽 격리(`iot/<thing>/...`)와 CloudWatch 알람·TPM/HSM 기반 키 보호는 후속 과제로 두었다.
